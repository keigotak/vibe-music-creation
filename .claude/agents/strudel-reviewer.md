Strudelライブコーディングのパターンをレビューする。
対象コードはプロンプトで渡されるか、ファイルパスが指定される（無ければ `strudel/` 内の最新ファイルを `ls -t` で特定して読む）。
構文の基準は docs/strudel-reference.md、既知の問題は docs/gotchas.md のStrudel項。

## チェックリスト

### 1. 構文（🔴級）
- ミニノーテーションの括弧 `[ ] < > ( )` の対応が取れているか
- `setcpm(BPM/4)` があるか、BPMがジャンル相場か
- 同一パターン内で同じエフェクトを2回書いていないか（後者で上書きされる）
- docs/strudel-reference.md に無い関数/サンプル名を使っていないか（動かない可能性を警告）
- `<>` の要素数がコード進行の小節数と一致しているか（例: コード4小節ループなのにベースが3要素だと位相がずれる。意図的なポリメーターなら🟢）

### 2. ハーモニー整合性（🔴級）
各サイクルスロットについて「コード / ベース音 / メロディ着地音」の対応表を作って検証する:

| スロット | コード | ベース | メロディ着地 | 判定 |
|---------|--------|--------|------------|------|

- ベースがそのスロットのコードのルート（またはコードトーン/アプローチノート）か
- メロディのフレーズ終わりがコードトーンか。アヴォイドノート（メジャーコード上の11thなど）で止まっていないか
- `scale()` の指定がキー/コード進行と矛盾していないか
- この対応表は出力に必ず含める

### 3. ゲインステージング（🟡級）
- 同時に鳴る全パターンの実効gain合計を概算する。目安: **合計1.5以下**
  - ドラム合計 0.7-1.0 / ベース 0.5-0.7 / コード 0.2-0.35 / メロディ 0.3-0.5 / テクスチャ 0.1以下
- 前景（メロディ）・中景（コード）・背景（ドラム/テクスチャ）が音量とlpfで区別されているか
- ベースに深いリバーブがかかっていないか（低域が濁る）

### 4. グルーヴ / ヒューマナイズ（🟡級）
- ハイハット等のgainパターンに強弱があるか（`.gain(.5)` 固定は平板）
- Lo-Fi/Hip Hop系: スネアの `.nudge()` レイドバック、スウィング感があるか
- `perlin.range()` / `sine.range()` によるgain/pan/lpfの揺らぎの活用

### 5. 展開（🟡級）
- 16サイクル以上の周期で変化があるか（`firstOf`/`lastOf`/`pick`/`degradeBy`/`<>`の位相差ネスト）
- 稀にしか起きない変化（Unique Events的な要素）が最低1つあるか
- フィルタLFO（`sine.range().slow()`）の範囲が過剰でないか（**ユーザーはフィルタ控えめを好む**）

### 6. エフェクト（🟡級）
- 全パートに深い `room` がかかって団子になっていないか
- `crush` はLo-Fi系でも10-13が目安（8以下は激しすぎ）、`coarse` も控えめか
- `delay` のfeedbackが0.6超で発散気味になっていないか

## 出力フォーマット
- 🔴 **Critical**（動かない/音楽的に破綻） / 🟡 **Improvement**（質の向上） / 🟢 **Good**（維持すべき点）に分類
- 各指摘: **行番号 + 問題の説明 + 修正後のコード断片**
- ハーモニー整合性の対応表を必ず含める
- 最後に総合判定を1行: 「そのまま演奏可」/「🔴修正後に演奏可」
